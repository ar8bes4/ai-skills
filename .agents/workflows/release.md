# リリース・ワークフロー (`/release`)

このワークフローは、現在のリポジトリの開発区切り（リリース）において、これまでの変更内容をまとめて `CHANGELOG.md` に追記し、新しいバージョンタグを打って GitHub に反映させるための手順書です。

### 前提条件
- 現在のディレクトリが対象の Git リポジトリであること。
- コミットされていない変更がないこと（必要であれば先にコミットしておくこと）。

### ステップ①：現在のバージョンと差分の確認
1. **現在のタグを確認**
   - `git describe --tags --abbrev=0` で直近のバージョンタグを取得します。
2. **直近の変更履歴（差分）を取得**
   - 直前のタグから現在の `HEAD` までのコミットログを `git log <直前のタグ>..HEAD --oneline` で取得します（初回は全てのログ）。
   - コミットメッセージから「何が追加されたか（Added）」「何が修正されたか（Fixed/Changed）」を分析します。

### ステップ②：新しいバージョンの決定
1. **バージョニングの提案**
   - 取得したコミットログに基づいて、システムがセマンティック・バージョニングに従い、次のバージョン番号を提案します。（例: `v1.0.1`, `v1.1.0`, `v2.0.0`）
   - **[ユーザー確認]** ユーザーに「次のバージョンは `vX.Y.Z` でよろしいですか？」と確認を取ります。

### ステップ③：CHANGELOG.md の更新
1. **Changelog の追記**
   - リポジトリのルートにある `CHANGELOG.md` を読み込みます。
   - 決定した新しいバージョン見出し（例: `## [v1.1.0] - $(Get-Date -f "yyyy-MM-dd")`）と、各変更内容（Added, Fixed など）のリストを先頭（既存のバージョンの上）に挿入します。
   - `write_to_file` や `multi_replace_file_content` を使ってファイルの中身を更新します。

### ステップ④：コミット、タグ付け、プッシュ
1. **一括反映（リリース処理）**
   - 更新された `CHANGELOG.md` をコミットします。
     - `git add CHANGELOG.md`
     - `git commit -m "chore(release): version <新バージョン>"`
   - 新しいタグを打ちます。
     - `git tag -a <新バージョン> -m "Release <新バージョン>"`
   - 変更とタグをリモートに一括プッシュします。
     - `git push origin HEAD`
     - `git push origin <新バージョン>`

### ステップ⑤：報告
- 完了後、「リリースが完了しました」というメッセージとともに、新しく生成した Changelog の内容をユーザーに共有します。
